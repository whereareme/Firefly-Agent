# 同行印记设置页设计

## 目标

在 Firefly 设置中增加“同行印记”页面，让 Firefly 直接管理独立运行的好感与关系 Sidecar。该页面只负责启停、进程生命周期、连接状态和必要配置，不承载关系内容本身。

Sidecar 继续保持独立工程和独立数据边界。Firefly 通过 OpenAI 兼容接口把模型请求交给 Sidecar，再由 Sidecar 转发至用户原本配置的模型服务。

## 非目标

- 不在 Firefly 设置页展示好感数值、关系阶段或趋势。
- 不展示或编辑记忆内容。
- 不展示礼物、纪念日或确认队列。
- 不把 Sidecar 源码合并进 Firefly。
- 不引入新的后台服务框架或第三方进程管理依赖。
- 不允许用户修改监听主机；主机固定为 `127.0.0.1`。

记忆确认、礼物和纪念日仍由 Sidecar 自带的 Tk 管理面板负责。

## 设置页信息架构

“同行印记”位于设置导航的“记忆回廊”之后，并使用与现有导航图标风格一致的新图标。页面沿用现有设置页的导航宽度、内容边距、字号、开关、按钮和状态样式，不建立新的卡片式视觉语言。

页面按以下顺序组织：

1. **状态**
   - 显示当前状态：已连接、正在启动、已停止、配置错误。
   - 显示当前 Sidecar 地址，例如 `http://127.0.0.1:8787/v1`。
   - 仅在存在故障时显示最近一条简短错误，避免输出完整日志或堆栈。
2. **运行控制**
   - 提供启动、停止、重新启动操作。
   - 按钮可用性随状态变化，避免重复启动或对已停止进程执行停止。
3. **连接配置**
   - 显示固定监听地址与端口配置。
   - 配置 Sidecar 工程路径、配置文件路径以及必要的启动参数。
   - 展示 Sidecar 将要转发的原始 OpenAI 兼容上游地址。
4. **自动接管**
   - 主开关控制“同行印记”是否启用。
   - 开启后由 Firefly 自动启动、重连并在退出时关闭 Sidecar。
   - 关闭后立即停止 Sidecar，并恢复 Firefly 原本的模型上游地址。
5. **诊断**
   - 提供精简的进程状态、连接检测结果和最近错误。
   - 不在设置页内实现完整日志查看器。

页面采用现有 `settingsWorkspace` 布局和既有间距体系，保持信息密度与其他设置面板一致。状态和操作放在首屏可见区域；低频配置与诊断位于下方。

## 组件边界

### 设置页

设置页负责收集配置、显示状态并发出启停命令。它不直接拼装进程命令、不自行轮询网络，也不拥有恢复上游地址的业务规则。

### Sidecar 进程控制器

新增一个小型、独立的 Qt 对象作为进程控制器，职责包括：

- 使用 PySide6 `QProcess` 启动和停止 Sidecar。
- 维护停止、启动中、运行中和错误状态。
- 接收 `started`、`finished`、`errorOccurred` 和标准错误输出信号。
- 对外发出状态变化和简短错误信息。
- 在启动前生成或更新 Sidecar 配置。
- 在应用退出时完成有界停止，必要时终止仍未退出的子进程。

控制器不读取或修改关系数据，也不解释 Sidecar 的业务响应。

### 配置接管协调

Firefly 保留“用户原始上游地址”和“Sidecar 本地地址”两个明确值。启用时保存原始上游并切换到 Sidecar；禁用或启动失败回滚时恢复原始上游。

该逻辑集中在单一协调入口，设置页、提供商切换和应用退出均调用同一套方法，避免不同路径产生不一致状态。

## QProcess 生命周期

### 启用

1. 验证当前模型提供商是否使用 OpenAI 兼容的 URL 配置。
2. 读取并保存当前原始上游 URL；若已经处于接管状态，不把 Sidecar 地址覆盖成新的“原始地址”。
3. 根据当前设置写入 Sidecar 配置。
4. 使用 `QProcess` 启动 Sidecar，并进入“正在启动”状态。
5. Sidecar 进程启动后进行连接检测。
6. 连接成功后，把 Firefly 模型地址切换为 `http://127.0.0.1:8787/v1`，重新应用运行时配置，并显示“已连接”。

只有连接检测成功后才完成流量切换，避免 Firefly 在 Sidecar 尚未可用时向空端口发送请求。

### 禁用

1. 先恢复已保存的原始上游 URL，并重新应用 Firefly 运行时配置。
2. 请求 Sidecar 正常退出。
3. 在超时后使用 `QProcess.terminate()`，仍未退出时再使用 `QProcess.kill()`。
4. 清除自动接管状态，但保留用户配置，方便下次启用。

恢复上游先于停止进程执行，以缩短模型服务不可用窗口。

### 手动控制

- “启动”只在功能已启用且进程停止时可用。
- “停止”先恢复原始上游，再停止当前进程，但不关闭主开关；界面明确显示已启用但已停止。
- “启动”连接成功后重新切换至 Sidecar 地址。
- “重新启动”先临时恢复原始上游，再生成配置并重启进程；连接成功后重新使用 Sidecar 地址。
- Firefly 不对用户在外部启动的未知 Sidecar 进程执行强制结束；端口被占用时报告配置错误，并保留原始上游连接。

### Firefly 启动与退出

- Firefly 启动完成后读取持久化开关；若已启用，自动启动 Sidecar，并在健康后接管上游。
- Firefly 退出时通过 `QApplication.aboutToQuit` 调用控制器关闭流程。
- 窗口的隐藏式 `closeEvent()` 不等于应用退出，因此不会误停 Sidecar。
- 退出流程必须有超时，不允许 Sidecar 阻塞 Firefly 关闭。

## Sidecar 配置生成

Firefly 写入 Sidecar 已支持的配置格式，不修改 Sidecar 源代码。生成内容至少包括：

- 固定主机 `127.0.0.1`。
- 用户配置的端口，默认 `8787`。
- 当前原始 OpenAI 兼容上游 URL。
- Sidecar 需要的模型、鉴权引用或启动模式字段。
- 独立数据目录和管理面板模式等现有 Sidecar 配置。

敏感信息遵循 Firefly 和 Sidecar 已有的凭据处理方式。设置页不回显完整密钥，诊断错误也不得包含密钥。配置写入采用完整结构化序列化和原子替换，避免部分写入产生损坏文件。

Sidecar 路径和配置路径使用绝对路径。启动时显式设置工作目录，避免 Firefly 从不同目录启动时找不到资源。

## 上游地址接管与恢复

启用状态持久化以下信息：

- 是否启用自动接管。
- Sidecar 端口和路径配置。
- 最后确认的原始上游 URL。
- 当前是否由 Sidecar 接管。

接管规则如下：

- 只有 OpenAI 兼容提供商可以启用。
- 当前地址若已是 `127.0.0.1:<sidecar-port>/v1`，不得再次保存为原始上游。
- Sidecar 未健康时不切换 Firefly 地址。
- 禁用、配置验证失败或启动失败时恢复原始上游。
- Firefly 非正常退出后再次启动时，若持久化状态显示接管中，则使用已保存的原始上游生成配置，而不是把本地 Sidecar 地址作为上游。

## 提供商变更

用户在“同行印记”启用期间保存新的模型提供商配置时：

1. 检查新提供商是否具备 OpenAI 兼容上游 URL。
2. 若兼容，将新地址保存为原始上游，更新 Sidecar 配置并重新启动 Sidecar。
3. Sidecar 恢复健康后继续让 Firefly 使用本地 Sidecar 地址。
4. 若不兼容，拒绝继续接管，停止 Sidecar、恢复新提供商的直接连接，并在“同行印记”页显示配置错误。

该流程接入现有模型设置保存路径，避免仅修改界面值却未同步运行时。

## 状态与错误处理

状态由进程状态和健康检查共同决定：

- **已连接**：进程正在运行且本地 OpenAI 兼容端点健康。
- **正在启动**：进程已请求启动，但健康检查尚未成功。
- **已停止**：进程未运行，且没有待处理的启动操作。
- **配置错误**：路径、端口、提供商兼容性、配置写入、进程启动或健康检查失败。

错误信息面向操作，例如“端口 8787 已被占用”“未找到 Sidecar 启动文件”“当前提供商不支持 OpenAI 兼容地址”。完整标准错误可以保留在应用日志中，但页面只显示最新的脱敏摘要。

健康检查采用有限次数、短间隔重试，并有总超时。超时后停止由 Firefly 启动的进程、恢复原始上游并进入配置错误状态，不进行无限自动重启。用户可修复配置后手动重启，或在下次 Firefly 启动时再次尝试。

## 持久化与兼容性

新增设置沿用 Firefly 现有配置存储机制，不创建第二套偏好数据库。缺少新字段时使用安全默认值：功能关闭、端口 `8787`、Sidecar 路径指向同级独立工程的预期位置。

旧配置首次打开时不会自动接管请求。只有用户明确开启后才保存原始上游并启动 Sidecar。

## 测试策略

### 控制器单元测试

- 生成的命令、工作目录和配置路径正确。
- 状态随 `QProcess` 信号正确转换。
- 正常停止、超时终止和强制结束路径正确。
- 错误摘要脱敏且不会暴露密钥。
- 健康检查成功、失败和超时均产生预期状态。

### 配置与接管测试

- 开启时保存原始上游，健康后切换到本地地址。
- 禁用和启动失败时恢复原始上游。
- 重复启用不会把本地地址保存为原始上游。
- 非正常退出后的恢复不会形成 Sidecar 自转发循环。
- 兼容提供商变更会更新配置并重启。
- 不兼容提供商变更会退出接管并给出配置错误。

### 设置页测试

- 导航顺序为“记忆回廊”之后的“同行印记”。
- 四种状态的文案和操作按钮可用性正确。
- 主开关、启动、停止和重启调用正确的控制器接口。
- 页面不展示好感、记忆、礼物或纪念日内容。
- 长路径、较长错误文本和不同窗口尺寸下无重叠或截断。

### 应用生命周期测试

- 启用配置下启动 Firefly 会自动启动 Sidecar。
- `aboutToQuit` 会停止由 Firefly 管理的 Sidecar。
- 仅隐藏聊天窗口不会停止 Sidecar。
- Sidecar 不可用时 Firefly 能恢复直接模型连接并正常退出。

### 回归验证

- 运行现有 Firefly 测试。
- 运行 Sidecar 现有测试，确认配置格式和启动方式未被破坏。
- 在 Windows 上进行一次真实联调：开启、连接、发送请求、重启、切换提供商、禁用和退出。

## 完成标准

- 用户可在 Firefly 设置中启用“同行印记”，无需手动启动 Sidecar。
- Sidecar 健康后 Firefly 请求经由本地网关转发，禁用后恢复原始上游。
- Firefly 能可靠显示连接状态和可操作的简短错误。
- 提供商变更、启动失败和应用退出不会留下错误上游地址或失控子进程。
- 设置页视觉和排版与现有设置界面一致，并严格保持管理功能边界。
